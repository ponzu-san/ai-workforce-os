後続検討メモ（パイプライン実走レビュー）
作成日: 2026-07-09
文脈: Claude API 接続後、design_only 等でパイプラインを通した際の気づき

1. 顧客向け成果物の見せ方
現状は社内向け Markdown 下書き＋承認のみ
顧客に見せる場合は Markdown 以外（整形ページ／PDF／表形式など）の可能性あり
社内ドラフトと顧客共有用は用途を分けて設計する
2. 成果物の直接編集
現状、中身修正の公式ルートはほぼ「却下」のみ
承認前に Markdown を直接編集できる機能を検討
編集後バージョン保存・履歴も合わせて設計
3. 却下以外の修正導線
コメント付き差し戻し → AI再生成／人間編集後に再承認、などの流れ
4. 指示の粒度と入力タイミング
「指示を保存」は実質ワークフロー開始前（営業）のみ
保存先は案件全体の Project Memory（工程IDなし）
工程ごとの指示として再設計するか検討
各工程の実行前にも指示追加できるようにする
5. 秘書AIと正式指示の役割分担
秘書チャットは secretary_chat 短期メモで、「保存済みの指示」一覧には入らない
相談用と正式指示を分ける
6. 契約工程の人間向け出力
契約も Markdown 社内下書きのみ
人間（顧客・関係者）向け出力形式を検討
7. 営業／契約の確認項目をチェックボックス選択
例: 「写真はこちらで用意」「ドメインはこちらで用意」
選択結果を契約工程の入力・チェックリストに反映
8. 承認コメントの可視化
コメントは DB approvals.comment に保存されるが、承認後UIに表示されない
承認履歴／工程画面／成果物画面への表示を検討
却下コメントを再実行・修正入力に使うかも検討
9. 要件定義成果物の末尾切れ調査
要件成果物の最後が切れていた
有力候補: planning の maxOutputTokens: 3000
詳細ページ／DB全文か、一覧プレビュー（200文字）だけの切りかも切り分け
10. プロンプト詳細ページの一括コピーボタン
プロンプト全文を一度にコピーできるボタン
11. デザインハンドオフの情報不足（ChatGPT検証）
現状の基本仕様は足りているが、生成AI一貫UIには不足。参考項目:

デザインコンセプト
ベンチマーク（参考サービス＋どの特徴）
レイアウトルール
レスポンシブ仕様
ダークモード
アイコン・イラスト
グラフデザイン
コンポーネント状態（Loading/Empty/Error等）
ページごとのレイアウト詳細
アニメーション
アクセシビリティ
デザイントークン
状態設計
実データ例
デザイン品質目標
→ Designer プロンプト／handoff テンプレ改善の参考

12. 外部成果物アップロード 1MB 制限
エラー: Body exceeded 1 MB limit（Server Actions 既定）
next.config.ts に serverActions.bodySizeLimit 未設定
対応候補: 上限引き上げ、クライアント事前チェック、大きい場合はURL登録案内
13. ファイル選択UI
type="file" がクリック可能に見えにくい
ボタン風ラベル等に改善
14. skip 工程のページ導線
FE/BE を skip したら該当工程ページは開かない／開けないUIを検討
15. design_only なのに QA が大量エラー＋画像が見えない
QA がフル開発前提（BE欠落等）で指摘している → テンプレ／skip に応じた QA 観点切替が必要
ファイル成果物はリンク表示のみ（画像プレビューなし）
1MB制限でアップロード失敗していた可能性も確認
16. QA／法務の参照ソース可視化
「何を見て」成果物を作ったか画面上で分からない
現状 allowlist: QA=要件・デザイン・FE・BE・外部開発／法務=要件・契約のみ（デザイン非参照）
実行時の参照一覧表示、入力ソース記録を検討
17. 納品画面
現状は Markdown 納品パッケージ下書き（サマリー／インデックス／引き継ぎ／サインオフ）
目的が分かりにくい
最終デザインファイル、コード／リポジトリ等の実成果物を見られるようにする
納品 Markdown の役割説明も明確化
18. 全体UX
ボタンを押しているだけで何が行われているか分からない
最終成果物が見えないままパイプライン完了した感覚
工程ごとの「今何をしているか／何ができたか」、完了時の最終成果物まとめ画面、確認してから完了の導線強化
